1. Problem & motivation

The catalog has shipped its build-phase work: 912 records across 34 sections, 85 cells per record, 15 analytical views, and three consumption surfaces (web app, MCP server, CLI). It is now an excellent reference.

It is not yet a recommender. Two questions from real users surface this gap:

Question typeExampleWhat's missing
Positioning "What model sits between Kimi-k2.6 and Opus 4.8 — good enough for scoped agentic tasks but cheaper than frontier?" No view places candidates between two named anchors on cost/capability axes.
Constraint matching "Which memory system + harness best fits my project — long-running, multi-agent, latency-sensitive?" No taxonomy of use-case suitability on memory rows; no surface that takes project constraints in and ranks candidates out.

The underlying data exists. The cost-economics and product-benchmark-matrix views already aggregate the relevant cells. What's missing is the decision surface on top — plus the normalized columns, sustainable cadence, and intake pipeline that keep recommendations defensible over time.

Framing Phase 2 turns the catalog from "browse and read" into "ask and rank." The catalog stays the source of truth; recommender surfaces are pure functions over it.

2. Goals & non-goals

Goals

  • Add three new column families to the schema: normalized cost, capability tier, use-case suitability tags.
  • Ship two recommender surfaces — positioning (between-anchors) and constraint-matching (project-shape) — backed by one shared ranking function.
  • Expose surfaces in all three consumption layers: web routes, MCP tools, CLI commands.
  • Establish a named analysis cadence (weekly recommendation-drift, quarterly capability-recalibration) backed by cron workflows that file structured findings.
  • Stand up a community submission pipeline with schema validation and mechanical auto-enrichment.

Non-goals

  • Not a "best model overall" leaderboard — rankings are always context-dependent, never globally ordered.
  • Not a replacement for the existing 15 analytical views — Phase 2 surfaces sit alongside them under web/src/routes/recommend/.
  • Not user accounts or personalization — every query is single-shot, stateless.
  • Not an LLM-inference recommender — ranking is keyword + structured-field matching, deterministic and auditable. LLMs may assist intake-bot enrichment but never run at query time.

3. Knowledge layer — schema additions

Three new column families. Each fits inside the existing record schema (currently 85 cells); the additions bring the count to ~95 cells. All new cells participate in Phase 1's staleness tracking (gate 6, cell-level visibility).

3.1 Column family A — Normalized cost

CellTypeNotes
cost_input_usd_per_mtoknumberPublic list price, vendor-reported. Null for self-hosted.
cost_output_usd_per_mtoknumberSame.
cost_tierenumfree | budget | mid | premium — derived bucket, recomputed nightly.
cost_pricing_modelenumper-token | per-request | subscription | self-hosted | free-tier
cost_last_verifieddateStaleness pipeline checks this against vendor pricing page.

Applicable to: foundation models, hosted memory services, agent platforms with metered pricing. Skip-eligible for research papers (decay_cause = research-complete) and self-hosted-only systems.

3.2 Column family B — Capability tier

CellTypeNotes
capability_composite_scorenumber 0–100Weighted composite across three task families: code, agentic reasoning, long-context. Recomputed quarterly.
capability_bandenumentry | competent | frontier — derived from composite.
capability_benchmark_sourcesarrayList of {benchmark, score, run_date, source_url} — transparency for the composite.
capability_last_verifieddateTied to quarterly recalibration run.

Benchmark source selection draws on the existing benchmark-trust view's vetted list, so we don't recreate the trust evaluation. The composite is not a blessed authority — it's a default starting point users can pivot from by inspecting the underlying scores.

Methodology The composite formula, normalisation, aggregation choice (trust-weighted arithmetic mean within family; geometric mean across families per the OECD/JRC handbook + UNDP HDI precedent), band cutoffs, and sensitivity checks are specified in docs/composite-methodology.md. Implementation: mcp/src/composite.ts, tests in mcp/src/test-composite.ts (40 assertions, all passing). Per-family sub-score cells (capability-code-score, capability-agentic-score, capability-longcontext-score) are spec'd in the methodology doc §11; the schema migration that adds them is its own PR.

3.3 Column family C — Use-case suitability

CellTypeNotes
use_case_tagsmulti-selectFrom controlled vocabulary below.
use_case_anti_tagsmulti-selectExplicit "do not use for" tags — equally valuable for ranking.

Initial controlled vocabulary

  • scoped-agentic — well-defined task, short horizon, single-tool sequences
  • long-running-session — multi-turn, persistent state, hours-to-days lifespan
  • multi-agent-coordination — shared context across agents, orchestration-heavy
  • memory-augmented-chat — single-user conversational, retrieval-focused
  • code-generation-focused — IDE / code-task primary use
  • analytical-summarization — long-context, factual extraction
  • latency-sensitive — sub-second response requirements
  • offline-capable — air-gapped or local-first deployment

Vocabulary is intentionally flat to start. Two thresholds govern when to revisit:

  • Monitor at 10% catalog membership — when any tag reaches 10% coverage, open a tracking issue and start collecting evidence on whether the tag has become too coarse.
  • Promote at 20% catalog membership — at this point a tag almost certainly bundles distinct sub-cases; split into a hierarchy.

Thresholds are deliberately conservative — earlier-than-needed taxonomy work costs less than carrying coarse tags that miscategorize submissions.

3.4 Cell-level provenance & verification

Every cell in the catalog has a known origin. The recommender's defensibility depends on being able to answer "where did this value come from, and has a human confirmed it?" for any cell that ends up in a ranking rationale.

Each record carries a parallel _provenance object keyed by cell name:

{
  "id": "kimi-k2-6",
  "capability_band": "frontier",
  "cost_input_usd_per_mtok": 0.15,
  "use_case_tags": ["scoped-agentic", "code-generation-focused"],

  "_provenance": {
    "capability_band":           { "source": "llm",        "verified": false, "generated_at": "2026-06-29" },
    "cost_input_usd_per_mtok":   { "source": "scrape",     "verified": true,  "scraped_at":    "2026-06-29", "scrape_url": "..." },
    "use_case_tags":             { "source": "human",      "verified": true,  "author":        "@maintainer", "edited_at": "2026-06-15" }
  }
}

Source values

  • human — authored or last-edited by a person via PR or web form.
  • scrape — mechanically derived (GitHub API, arXiv API, vendor pricing page extraction).
  • llm — suggested by an LLM during intake enrichment.
  • legacy — predates the provenance model; assigned during Gate 1's existing-cell backfill. Treated as verified-by-default since the cells were trusted catalog state before Phase 2 began.

Verification rules

  • Every cell is fillable by LLM enrichment if a value cannot be obtained mechanically. No cell is off-limits — including cost cells where vendor pricing pages are unparseable.
  • LLM-generated cells are always labeled. The source: "llm" marker is visible in every surface that displays the cell — web UI shows a "LLM, unverified" badge; MCP tool responses include it in the caveats array; CLI prints it inline.
  • Unverified LLM cells are non-load-bearing for ranking. rank_candidates() excludes unverified LLM values from score calculation — they appear in the candidate's profile but cannot influence position. The rationale string explicitly notes when a cell was skipped for this reason.
  • Verification flips the bit, not the value. A human reviewing an LLM-generated cell either edits it (changing source to human) or sets verified: true with their handle. Verified-LLM cells become load-bearing.
  • Scraped cells are verified-by-default. Mechanical extraction with a stored scrape_url is treated as load-bearing without separate human sign-off; staleness pipeline catches drift.
Why this matters Without provenance, an LLM-generated cell looks identical to a human-curated one in the data file. The recommender would unknowingly cite hallucinations as facts. The provenance model makes "this is a guess" a first-class data property, surfaces it everywhere, and excludes it from rankings until a human says otherwise.

3.5 Backfill plan

Gate 1 owns the backfill end-to-end. The gate does not close on empty cells.

  1. Schema migration: additive cells with null defaults across all 912 rows.
  2. Legacy-cell provenance backfill: every existing cell (the 85 pre-Phase-2 cells) gets a _provenance entry with source: "legacy", verified: true. This is a one-shot, mechanical operation — no judgment required. Where git history clearly identifies the writing script (e.g. scripts/fetch_commit_trajectories.py for download-trajectory cells), source is upgraded to scrape with the script name recorded.
  3. Mechanical fill of Phase 2 cells:
    • Cost cells: vendor pricing-page scrape for every hosted model / service in the catalog (not just top 50). Failures fall through to LLM enrichment with verified: false.
    • capability_benchmark_sources: pulled from existing benchmark-trust view sources where rows are already linked.
  4. LLM judgment fill of Phase 2 cells: capability_band, capability_composite_score, and use_case_tags filled via /loop deep-fill pattern (sub-agent per column class, CSV apply via /tmp script, comma-delimited). All marked source: "llm", verified: false — non-load-bearing until human review.
  5. Intake template extension: new submissions required to populate the new columns or explicitly mark not-applicable; Gate 7 owns the intake-side enforcement.

End-state on Gate 1 close: every row in data/landscape.json has a populated _provenance object covering all cells (legacy + Phase 2), with Phase 2 cells filled by the best available source (scrape > LLM-unverified > null where genuinely not applicable).

4. Capability layer — recommender surfaces

4.1 Shared core: rank_candidates()

Single pure function over the catalog. Surface code calls this; the surface differences are in input transformation and output presentation only.

rank_candidates(
  constraints: ConstraintSet,
  anchors?: { low: RecordId, high: RecordId },
  k: number = 5
) -> Array<{
  record: Record,
  score: number,            // 0..1
  rationale: string[],      // bulleted reasons it ranked here
  caveats: string[],        // explicit warnings (stale data, missing benchmarks, etc.)
}>

Implementation follows the existing shared-analysis pattern in this repo (see web/src/lib/analyses/citation-prediction.ts and its mcp/src/ mirror):

  • Single source of truth: web/src/lib/analyses/recommender.ts
  • Verbatim mirror for npm package: mcp/src/recommender.ts (required because mcp/ ships as a standalone npm package with its own rootDir)
  • CLI consumption: imports from ../../mcp/dist/recommender.js (matches existing CLI pattern in cli/src/landscape.ts)

No model inference, no network calls — fully deterministic. Web, MCP, and CLI must return identical rankings for identical inputs.

4.2 Surface 1: Positioning recommender

Route/recommend/between
MCP toolbetween_models(anchor_low_id, anchor_high_id, use_case?)
CLIlandscape recommend between <id> <id> [--use-case TAG]
InputTwo anchor record IDs, optional use-case tag filter.
OutputRanked candidates whose cost × capability_composite_score place them between the two anchors, with a one-line "why this fits" rationale.
Web UXTwo slot pickers + a scatter plot of cost vs. capability with anchors highlighted and candidates labeled.

4.3 Surface 2: Constraint matcher

Route/recommend/by-constraints
MCP toolrecommend_for_project(description, max_results=5)
CLIlandscape recommend for "<project description>"
InputFree-text project description or structured form: project_shape, scale, latency_budget, persistence requirements, deployment mode.
OutputRanked memory + harness candidates with matched use-case tags highlighted and anti-tags flagged as caveats.
Web UXTextarea + toggleable structured form; results split by category (memory / harness / model) with cross-links.

Free-text input is parsed into a ConstraintSet via keyword matching against the controlled vocabulary — not via LLM. This keeps the recommender auditable and reproducible.

5. Analysis cadence

All recurring analysis runs on a daily floor. The model release cycle is now measured in days, not quarters — a quarterly recalibration would mean three months of stale recommendations between updates. Cron is cheap; correctness decay is expensive. Every workflow below fires daily unless its input data fundamentally cannot move that fast.

Cadence principle Default to daily. Move slower only when input data physically cannot change faster (e.g. peer-reviewed benchmark publications). Move faster than daily when triggered by external events (new model release, vendor pricing change).
CadenceWorkflowWhat it doesOutput
Daily staleness.yml (existing — Phase 1) Flags rows with stale cost/capability cells. Cell-level after gate 6. GitHub issues with stale-row / stale-cell labels.
Daily recommendation-drift.yml (new) Re-runs the canonical-questions set (~20 entries, incl. "between Kimi-k2.6 and Opus 4.8 for scoped agentic"). Diffs the top-5 against the prior 24h snapshot. Opens issue on any top-5 movement. docs/drift-YYYY-MM-DD.md + GitHub issue if movement detected.
Daily capability-incremental.yml (new) Re-scores capability_composite_score for any row whose capability_benchmark_sources changed in the past 24h, or whose underlying benchmark publication updated. Cheap — only touches changed rows. Auto-PR per affected row with score diff + rationale.
Daily catalog-pulse.yml (new) Aggregate health snapshot: new entries, retirements, top-N most-queried recommendation surfaces, drift hot-spots. Posts to /findings. Public daily card on /findings + last-7-days rollup link.
Daily vendor-price-sweep.yml (new) Scrapes vendor pricing pages for hosted services in the catalog. Diffs against stored cost_input_usd_per_mtok / cost_output_usd_per_mtok. Auto-PR on any change. Auto-PR per price change + issue if the change crosses a cost_tier boundary.
Weekly capability-rebaseline.yml (new) Full re-score across all 912 records — anchors the daily incremental updates, catches drift the incremental pass missed (e.g. a benchmark source quietly revising methodology). docs/FINDINGS-YYYY-MM-DD.md with movement summary + tracking issue.
On release model-release-watch.yml (new — event-driven, not cron) Webhook + RSS poller for known vendor announcement channels (OpenAI, Anthropic, Google, Mistral, DeepSeek, Moonshot, etc.). Fires within minutes of a detected release. Opens an intake-pending issue with auto-enriched stub. Same-day catalog entry for major releases — no waiting for the next daily sweep.
Curated questions The daily drift workflow needs a small list (~20) of real questions worth tracking. Initial list seeded from real user questions (the Kimi/Opus question is question #1) — list lives at docs/canonical-questions.yml and grows by PR. Cap at ~50 entries to keep daily runs cheap.
Tradeoff Daily cron amplifies any bug in the ranking function — a wrong rationale gets republished daily until caught. Mitigation: every daily-run output is auto-PR or auto-issue, never a direct write to data/landscape.json. Maintainer review remains the gate for catalog-state changes; daily runs change signal, not state.

6. Submission pipeline

The catalog's long-term value depends on submissions being easy and quality being defensible. Three entry points, all converging to the same auto-PR flow.

6.1 Entry points

  1. GitHub issue template — submit-record.md
    Required: record name, category, primary URL, brief description.
    Optional: pricing, benchmark scores, use-case tags.
    Auto-labeled intake-pending; triggers intake-research workflow.
  2. Submission form on /submit web route
    Same schema. Submits via GitHub API (requires GitHub auth to prevent spam — see open questions).
    Lower friction for non-developer contributors.
  3. Pull request — direct schema edit
    For maintainers and trusted contributors. Same schema validation in CI.

6.2 Pipeline

  1. Submission lands → intake-research.yml triggered (existing cron, extended).
  2. Bot runs schema validation: required cells present, primary URL resolves, no duplicate ID.
  3. Bot runs auto-enrichment:
    • GitHub URL → fetch stars, last-commit, language, license.
    • arXiv URL → fetch title, authors, published date, abstract.
    • Vendor pricing page URL → attempt structured-extract of cost cells.
    • (Optional, flagged) LLM-suggested use-case tags + capability-band guess, all marked bot-suggested and requiring sign-off.
  4. Bot opens auto-PR with stub row pre-filled with mechanically-derivable cells.
  5. Maintainer reviews subjective cells (capability_band, use_case_tags, cost_tier).
  6. On merge → next staleness/audit run picks up the new row.

6.3 Quality gates on submissions

  • Hard reject: duplicate ID, broken primary URL, missing required cells.
  • Soft warning (PR opens with a flag, not blocked): row contains many LLM-source cells with verified: false. These cells will display as unverified everywhere and are excluded from ranking until human-reviewed (per §3.4) — the row still merges so the catalog isn't starved of entries.
  • Auto-merge eligible: row's only LLM-source cells are verified: false AND every required mechanical cell is populated. The row becomes immediately visible but with reduced ranking weight until cleanup.

This design lets contributors land partial rows without maintainer bottleneck — the provenance model handles the trust signal automatically. Maintainer review focuses on flipping verified: true on LLM cells, not on whether the row gets merged at all.

8. Resolved design questions

All five design questions are settled. Resolutions captured below; each links to the section of the spec where the rule lives.

Q1: Composite capability score — single number or per-task only?

Resolved: ship the composite as default-on, with per-task scores always exposed in the UI and MCP-tool response. Don't dilute granularity; don't over-process when a composite suffices. See §3.2.

Q2: Use-case tag vocabulary — flat or hierarchical?

Resolved: start flat. Trigger thresholds lowered for earlier signals:

  • Monitor at 10% catalog membership — open tracking issue, gather evidence.
  • Promote at 20% — split the tag into a hierarchy.

See §3.3.

Q3: Submission spam mitigation — GitHub auth or rate-limit?

Resolved: GitHub auth required on the /submit route. Reasonable barrier, piggybacks on existing identity. Anonymous submissions stay possible via the issue template (covered by GitHub's own anti-spam). See §6.1.

Q4: LLM-assisted intake enrichment — on or off?

Resolved: on, universally, with provenance tracked per cell.

  • LLM enrichment fills any cell it can usefully fill — including cost cells, when mechanical scrape fails.
  • Every LLM-generated cell is labeled source: "llm" with verified: true | false.
  • Unverified LLM cells are non-load-bearing: visible everywhere, but excluded from ranking score calculation and flagged as "incomplete" in surface output.
  • Verification is a human flipping the bit (or editing the value, which changes source to human) — there is no automatic verification.

This is more aggressive than the original recommendation (which excluded cost cells). The provenance model in §3.4 makes that safe: a wrong LLM value can't pollute rankings, only the user-visible profile.

Q5: Canonical-questions seed list — who curates?

Resolved: two-track approach.

  • Track 1 — research sprint (Gate 5): structured sweep of external sources before the daily drift cron goes live. Targets: GitHub Discussions, Discord, public AI forums (r/LocalLLaMA, r/MachineLearning, Hacker News threads on model releases), model launch comment threads, known practitioner Twitter accounts. Goal: ~50 categorized questions, not 20.
  • Track 2 — ongoing curation: PRs to add canonical questions accepted from anyone post-launch; removal requires maintainer approval so the drift signal stays stable.

See Gate 5 in §7.
