Trajectory — what's used today vs what's growing or dying

Six panels. The first five combine funding cadence, release recency, GitHub stars, mindshare/citations, inbound integration edges, and lineage membership into a heuristic velocity story per record — confidence labels are surfaced everywhere so a reader can weigh each callout. The sixth fits a BIMATEM-style logistic S-curve to every row with enough temporal signal and reads pre-growth / growth / saturation / decline straight off the inflection — quantitative complement to the heuristic buckets, and the math the Gartner Hype Cycle waves at without ever publishing.

How rows are bucketed

Accelerating
Recent funding (≤12mo) or newest member of a detected lineage or ≥10k GH stars or mindshare ≥70.
Steady
Inbound integrations/citations present and no decay signal — cited and maintained but not currently accelerating.
Decelerating
Two of: funding ≥18mo old · last release ≥18mo old · last commit ≥18mo old. Slope down but not yet flatlined.
Dead
Cross-references survivorship: any record the survivorship view labels Abandoned (no release/commit in 24+mo) routes here.
Confidence
High = ≥2 corroborating signals. Medium = exactly one strong signal. Low = ambiguous or single weak signal.

Panel 6 methodology — BIMATEM-style logistic fit

For each row with at least 5 dated observation points spanning at least 12 months, we fit the logistic y = L / (1 + exp(-k · (t − t0))) via coarse grid search over (L, k, t0) followed by three local refinement passes. The framework follows the BIMATEM bibliometric tradition (Bengisu & Nekhili 2006 onwards) and the citation-trajectory work in Wang, Song & Barabási Science 2013 — both fit a 3-parameter logistic to cumulative event counts and read maturity off the inflection point.

Pre-growth
Observation time below t0 − 2σ where σ = 2/k. Below ~12% of estimated carrying capacity.
Growth
t0 − 2σ ≤ t < t0. Steepest part of the ramp; saturation date predicted as t0.
Saturation
t0 ≤ t < t0 + 2σ. Post-inflection; rate is slowing toward the plateau.
Decline
Past saturation AND the most-recent 2-of-3 observations sit below 95 % of the fitted curve — distinguishes a real fall-off from a long plateau tail.
Insufficient data
Fewer than 5 distinct cumulative-count anchors or under 12 months of observable history. We refuse to force a 3-parameter fit on too little signal; the bias toward this bucket is a property of catalog-level data, not the underlying technology.
Signal sources
Priority: (1) commit trajectory — real monthly cumulative commit counts from the GitHub Commits API, populated by make refresh-commit-trajectories (T3-prep-1 / issue #50; the strongest signal we have for product / OSS rows because it is real per-month data, not a synthesised piecewise reconstruction); (2) citation trajectory — real yearly cumulative within-catalog inbound-citation counts bucketed offline from the S2 reference cache by make bucket-citations (T3-prep-2 / issue #51; the strongest signal we have for academic-paper rows when ≥3 distinct years of inbound citations exist); (3) download trajectory — real monthly cumulative NPM / PyPI download counts, populated by make refresh-download-trajectories (T3-prep-3 / issue #52; cleanest adoption signal for OSS library / SDK rows that publish to a package registry — smooth monotonic curves that fit the logistic well); (4) citations + per-year rate — synthesised piecewise fallback for paper rows without a citation-trajectory cell; (5) dated milestones across created / latest-release / funding / claims / customers / api-surface / compliance / pricing / deployment cells for products; (6) star-growth on OSS repos with +N/mo metadata.
